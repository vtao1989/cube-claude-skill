# MPICORES case studies

The first thing the user must know is that the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter only provides parallel capabilities when
used on a problem with multiple user classes, as this parameter essentially
allows the number of user classes defined by the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter to be run simultaneously. We will again
examine our three case studies here; performance results for using the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter on these are provided in Table 4.

the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter does a much better job of speeding up
the solution time for the two-core case, but then does very little from two to
four cores. In this case we find the initial benefit from executing the large
portions of sequential code for each class in parallel, but also find a
limitation of using the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) when increasing cores. First, we note that though
the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter may have been set to 4, the program
reduced it to three cores because the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameter can only run user classes in parallel
and we only have three for this problem. The primary factory in only achieving
an extra 3-second reduction when running on the extra MPICORE in this case is
that the program can only run as fast as its slowest user class will let it. In
this case we find that a single user class takes the full 90 seconds to run on
its own on a single core, and so the program cannot perform any better than
this without increasing the
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html) parameter.

Case 2: Medium-Size Dynamic Problem
(see Table 4) â The second test
case again shows good parallel results, and better with two
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) than two
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html). Indeed, we see that the two user classes being run
by this problem take nearly the same amount of time hence providing a good
balance that allows the performance increase to approach near perfect speedup
of 2X. However, since we only have two user classes for this problem, we cannot
achieve with
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) the speed provided by using more
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html).

Case 3: Large-Size Static Problem (see
Table 4) â We find that our large static test case does not perform
nearly as well by increasing the
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html)value as we found by increasing the
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html)parameter. The large amount of time this problem spends in the
optimization loop leads to the belief that we should see good overall
performance with the
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html) parameter but we would still expect to see good run
time reduction by increasing
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) considering the large number of user classes to be
estimated. We see, however, that this is not necessarily so. The reason for
this falls back to the lesson we learned above in Test Case 1 where we found
our problem could only run as fast as our slowest running user class, and in
this case we see that a single user class runs in the 85000 second range.

Two main points can be made from an examination of these
test cases:

1. The use of
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) allows user classes to be run in parallel, which
can lead to the best speedups since sequential code sections are performed in
parallel.

2. A problem will only run as fast as its slowest running
user class.

Having examined the use of both the
[NCORES](GUID-E23E89B8-1CBA-41D1-A643-788A7C2CC72B.html) and
[MPICORES](GUID-539DD601-CF95-495D-8993-D058970BF046.html) parameters individually, we will now move to
examine how they can be used together to provide the best possible performance
gains. See next section,
